# 5. Signing Pipeline

> **Current state:** the Gradle **release signing config exists and works**
> (`android-app/build.gradle.kts` — `signingConfigs["release"]`, wired to the
> `release` build type). It reads credentials from environment variables (CI) or
> a git-ignored `android-app/keystore.properties` (developers); if no keystore is
> configured it produces an **unsigned** release APK so the build still runs.
> `.github/workflows/release.yml` performs the signed build + `apksigner`
> verification on a `v*` tag. What is **not** in the repo (correctly) is the
> release **keystore itself** and its passwords — those are secrets. The shell
> backend is delivered as a source tarball and is integrity-protected with
> checksums + GPG rather than code-signed (§ 5.3).

## 5.1 What actually needs signing

| Artifact | Signing today | Signing target |
|----------|---------------|----------------|
| Android APK (`android-app/`) | Debug key only (implicit) | **Release keystore** (v2/v3 APK Signature Scheme) |
| Source tarball `ANDRAX-2.0.tar.gz` | Not signed | **GPG detached signature** + SHA-256 checksum |
| Magisk module zip (`magisk-module/`) | Not signed | SHA-256 checksum (Magisk verifies its own zip) |

## 5.2 APK signing model

Android requires every installed APK to be signed. Two keys are in play:

* **Debug key** — auto-generated by the Android SDK (`~/.android/debug.keystore`),
  used by `assembleDebug`/`installDebug`. Fine for development, **never** for
  distribution.
* **Release ("upload") key** — a keystore *you* create and guard. Every public
  release APK must be signed with the **same** release key, forever: Android ties
  app-update identity to the signing key, so losing it or rotating it (outside
  Play App Signing key rotation) means users cannot upgrade in place.

### One-time: create the release keystore

```sh
keytool -genkeypair -v \
  -keystore andrax-release.keystore \
  -alias andrax \
  -keyalg RSA -keysize 4096 -validity 10000 \
  -storetype PKCS12
```

Store the keystore and its passwords **outside the repository**. Recommended:
a password manager / secrets vault, and — for CI — GitHub encrypted secrets
(§ 5.4). Add belt-and-suspenders ignores:

```gitignore
# .gitignore
*.keystore
*.jks
android-app/keystore.properties
release.keystore
```

### Gradle release signing config

Add to `android-app/build.gradle.kts` (extending the module file in
`build-notes.md`). Read credentials from the environment (CI) or a local,
git-ignored `keystore.properties` (developers):

```kotlin
import java.util.Properties
import java.io.FileInputStream

val keystoreProps = Properties().apply {
    val f = rootProject.file("android-app/keystore.properties")
    if (f.exists()) load(FileInputStream(f))
}
fun cred(key: String, env: String) =
    (System.getenv(env) ?: keystoreProps.getProperty(key))

android {
    signingConfigs {
        create("release") {
            storeFile = file(cred("storeFile", "ANDRAX_KEYSTORE_FILE") ?: "release.keystore")
            storePassword = cred("storePassword", "ANDRAX_KEYSTORE_PASS")
            keyAlias = cred("keyAlias", "ANDRAX_KEY_ALIAS")
            keyPassword = cred("keyPassword", "ANDRAX_KEY_PASS")
        }
    }
    buildTypes {
        getByName("release") {
            isMinifyEnabled = false
            signingConfig = signingConfigs.getByName("release")
        }
    }
}
```

Local `keystore.properties` (git-ignored):

```properties
storeFile=/absolute/path/andrax-release.keystore
storePassword=…
keyAlias=andrax
keyPassword=…
```

Then `./gradlew assembleRelease` produces a signed release APK under
`android-app/build/outputs/apk/release/`.

### Verify the signature

```sh
apksigner verify --verbose --print-certs app-release.apk
# confirm SHA-256 of the signing cert matches your known release cert
keytool -printcert -jarfile app-release.apk
```

## 5.3 Source tarball & module integrity

The backend ships as source, so integrity is provided by checksums + GPG rather
than code signing:

```sh
# Build the tarball (see Release instructions §)
git archive --format=tar.gz --prefix=ANDRAX-2.0/ -o ANDRAX-2.0.tar.gz HEAD

# Checksum + detached GPG signature
sha256sum ANDRAX-2.0.tar.gz > ANDRAX-2.0.tar.gz.sha256
gpg --armor --detach-sign ANDRAX-2.0.tar.gz     # → ANDRAX-2.0.tar.gz.asc
```

Publish `.sha256` and `.asc` alongside the tarball on the GitHub Release.
Consumers verify with `sha256sum -c` and `gpg --verify`.

## 5.4 Signing in CI

Never commit a keystore. In CI, provide it as encrypted secrets and materialize
it only inside the release job:

| Secret | Contents |
|--------|----------|
| `ANDRAX_KEYSTORE_B64` | `base64 -w0 andrax-release.keystore` |
| `ANDRAX_KEYSTORE_PASS` | store password |
| `ANDRAX_KEY_ALIAS` | `andrax` |
| `ANDRAX_KEY_PASS` | key password |
| `ANDRAX_GPG_PRIVATE_KEY` | ASCII-armored GPG private key (for tarball signing) |
| `ANDRAX_GPG_PASSPHRASE` | GPG passphrase |

The release job decodes the keystore to disk, runs `assembleRelease`, signs the
tarball, then the runner is destroyed. See the `release.yml` skeleton in
[CI/CD § B.2](04-cicd-pipeline.md#b2-releaseyml--build--publish-on-a-tag).

## 5.5 Key management rules

1. **One release key, kept forever.** Back it up in at least two secure
   locations. Losing it breaks in-place updates for all users.
2. **Passwords live in a secrets manager**, never in the repo, never in shell
   history (use a `keystore.properties` file or env vars).
3. **Debug ≠ release.** Only `assembleRelease` output is distributable.
4. **Publish the release cert fingerprint** (SHA-256) in the repo so users can
   pin/verify what a legitimate ANDRAX 2.0 APK is signed with.
5. **Consider Play App Signing** if you ever distribute via Google Play — it
   lets Google hold the app signing key and enables key rotation, while you keep
   only an upload key.
